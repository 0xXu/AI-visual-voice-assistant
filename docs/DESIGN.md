# AI 视觉语音助手设计与实现复盘

## 1. 文档目的

本文基于项目现有设计规格、实施计划、当前代码和测试，对首版设计与实际结果进行复盘，重点回答两个问题：

1. 项目计划实现哪些用户故事，最终实现了哪些。
2. 项目考虑过哪些控制运营成本的技巧，实际采用了哪些。

本文所说的“运营成本”主要指 Gemini Live API 的模型与 token 消耗、实时音视频传输带宽，以及长时间或异常会话占用的后端资源。浏览器端计算和部署复杂度也作为次要成本纳入考虑。

## 2. 产品目标与范围

项目的目标是提供一个浏览器端实时视觉语音助手。用户授权摄像头和麦克风后，可以通过语音或文字询问当前画面中的物体、环境或操作步骤，并获得 AI 的实时语音和文字回答。

首版以比赛演示和单用户实时视觉问答为核心，采用以下范围约束：

- 前端使用 React、TypeScript 和 Vite，负责设备采集、界面状态和媒体播放。
- 后端使用 FastAPI 和 Gemini Live API，负责协议校验、输入调度、模型连接、会话限制与用量统计。
- Gemini API Key 仅保存在后端。
- 首版不实现登录、云端历史记录、多人会话、文件上传、屏幕共享、工具调用、网页搜索或离线模型。

## 3. 用户故事：计划与最终实现

### 3.1 实现情况总览

| 用户故事 | 原计划 | 最终结果 | 说明 |
|---|---|---|---|

| 明确展示会话状态 | 首版必须实现 | 已实现 | 展示连接、聆听、思考、回答、恢复和结束状态 |

| 文件上传、屏幕共享、工具调用和搜索 | 首版不实现 | 未实现，符合范围 | 保持首版实时视觉问答的范围可控 |

### 3.2 最终完成的核心用户流程

最终实现的完整流程为：

1. 用户从入口页主动开始，不会在页面加载时自动请求权限或创建 Gemini 会话。
2. 用户在设备检测页查看摄像头预览和麦克风状态，并选择输入设备。
3. 用户确认后进入实时会话；前端收到后端 `connected` 状态后才开始发送媒体。
4. 前端持续发送小块音频和低频 JPEG 画面，用户也可从记录抽屉发送文字。
5. AI 返回语音和文字内容，前端实时播放并展示转写。
6. 用户可打断 AI、静音、暂停画面、查看记录或结束会话。
7. 网络或 Gemini 连接需要迁移时，系统进行有限恢复，并展示恢复状态。
8. 会话因用户操作、空闲、最长时限或预算结束后，前端等待最终用量数据并展示结果页。

### 3.3 与原计划存在的差异

主要差异是实时翻转镜头没有完整落地。当前设备检测页可以选择摄像头，但实时页面的翻转操作仍为空实现。因此，这一用户故事只能认定为部分完成。

原设计还考虑通过画面变化程度动态降低采样频率。最终版本实现了固定低帧率、网络拥塞跳帧和页面隐藏暂停，但没有实现浏览器端视觉变化检测。这是有意的范围取舍，而不是遗漏：首版不值得为少量潜在帧节省引入额外算法、CPU 开销和测试复杂度。

## 4. 运营成本控制：候选技巧

设计阶段考虑的成本控制方法可以分为五类。

### 4.1 会话生命周期控制

- 只有用户显式点击开始后才创建 Gemini Live 会话。
- 用户无有效输入一段时间后自动结束会话。
- 限制单次逻辑会话的最长持续时间。
- 设置单会话 token 预算，到达阈值后停止继续消耗。
- 浏览器断开或任务异常时及时清理模型连接和后台任务。
- 对重连次数设置上限，避免异常状态下无限重试。

### 4.2 视觉输入控制

- 降低图片尺寸、JPEG 质量和采样帧率。
- 在画面稳定时降低帧率或停止上传。
- 页面进入后台时暂停抽帧。
- 用户暂停摄像头时停止上传。
- 网络发送缓冲过大时直接跳过当前帧。
- 不建立视频帧队列，只保留最新画面。
- 后端拒绝过大、过期、重复或倒退序列的图片。
- 使用模型的低媒体分辨率配置。

### 4.3 音频和文本输入控制

- 音频采用 20 至 40 毫秒的小块传输，限制单块大小。
- 音频与文本使用有界队列，避免流量高峰无限占用内存。
- 对拥塞输入施加背压，而不是无限缓存。
- 限制单条文字问题的字符数。
- 使用云端自动语音活动检测，避免额外维护本地 VAD。

### 4.4 模型上下文与路由控制

- 为长会话启用滑动窗口上下文压缩。
- 使用偏低成本、适合实时交互的 Flash Live 模型。
- 考虑按任务复杂度动态路由不同模型。
- 考虑先在本地判断是否需要上传画面。
- 考虑使用本地视觉变化检测减少无效帧。

### 4.5 用量可观测性

- 统计音频字节、文字字符、图片帧数和图片字节数。
- 区分已接受图片和被新帧替换的图片。
- 记录输入、输出和总 token。
- 记录会话持续时间和首响应延迟。
- 在结果页展示关键用量，使演示者能发现异常消耗。

## 5. 实际采用的成本控制措施

### 5.1 已采用

| 措施 | 实际实现 |
|---|---|
| 按需创建云会话 | 只有收到用户触发的 `start_session` 后才连接 Gemini |
| 空闲自动结束 | 默认连续 45 秒没有通过校验且被调度器接受的输入时结束 |
| 最长会话时限 | 默认单个逻辑会话最多 600 秒 |
| token 预算 | 默认累计 50,000 token 后以 `budget_exceeded` 结束 |
| 有限恢复 | Gemini 恢复和无句柄回退各有次数限制；浏览器 WebSocket 也只进行有限重连 |
| 低成本模型配置 | 默认使用 `gemini-3.1-flash-live-preview` |
| 低媒体分辨率 | Gemini Live 配置固定使用 `LOW` 媒体分辨率 |
| 上下文压缩 | 启用滑动窗口 context compression |
| 低频视频采样 | 前端默认约每秒采集一帧 |
| 图片缩放 | 图片长边最多 960 像素 |
| 分级 JPEG 压缩 | 依次尝试约 0.68、0.55 和 0.42 的质量，超过 512 KiB 则丢弃 |
| 后台暂停采样 | 页面不可见时不采集图片 |
| 网络拥塞跳帧 | `WebSocket.bufferedAmount` 超过 256 KiB 时跳过当前帧 |
| 用户暂停画面 | 暂停后不继续上传视频帧 |
| latest-only 视频 | 后端只保留序列最新的待发送画面，新帧替换旧帧 |
| 媒体大小校验 | 音频默认不超过 8,192 字节，图片默认不超过 524,288 字节 |
| 过期帧过滤 | 图片时间戳默认只能与后端时间相差 2,000 毫秒 |
| 有界输入队列 | 音频队列默认 32，文本队列默认 8 |
| 输入背压 | 队列拥塞时限制等待提交数并返回错误，不无限积压 |
| 文字长度限制 | 单条文字默认最多 2,000 字符 |
| 用量统计 | 最终汇总媒体量、token、持续时间和首响应延迟 |
| 可靠清理 | 会话关闭时排空或取消调度任务，并设置清理硬超时 |

这些措施同时作用于成本和稳定性。例如 latest-only 视频既减少无效视觉输入，也避免旧画面排队导致模型看到过时场景；有界队列既限制内存，也避免积压数据在网络恢复后集中发送并产生额外模型消耗。

### 5.2 想到但未采用

| 技巧 | 未采用原因 |
|---|---|
| 浏览器端画面变化检测 | 会增加前端算法、CPU 与测试成本；首版固定低帧率和拥塞跳帧已能覆盖主要风险 |
| 动态模型路由 | 当前实时会话只配置一个 Gemini Live 模型，路由会增加会话管理与效果评估复杂度 |
| 本地语音活动检测替代云端 VAD | 可能影响自然打断和语音边界判断，还需处理不同设备与噪声环境 |
| 浏览器端模型或离线视觉理解 | 会显著增加包体、设备要求和兼容性成本，不符合演示优先目标 |
| 客户端直接连接 Gemini | 虽可能减少后端转发，但会暴露 API Key，并削弱统一预算、校验和审计能力 |
| 无限自动重试 | 可能在服务异常时持续占用资源或产生重复会话，因此明确限制恢复次数 |
| 云端保存完整媒体和对话历史 | 会增加存储、隐私和合规成本，首版只保留当前浏览器内的文本状态 |

## 6. 成本控制的数据流

```text
浏览器设备
  ├── 16 kHz PCM16 小音频块
  ├── 约 1 fps、最长边 960 px 的 JPEG
  └── 最多 2,000 字符的文字
          │
          ▼
前端发送保护
  ├── 页面隐藏时暂停视频
  ├── 网络缓冲过大时跳帧
  ├── 暂停摄像头时停止上传
  └── 图片过大时降低质量或丢弃
          │
          ▼
后端协议与调度保护
  ├── 严格校验大小、格式、时间戳和序列
  ├── 音频与文本有界排队
  ├── 视频仅保留最新帧
  └── 拥塞时施加背压
          │
          ▼
Gemini Live
  ├── Flash Live 模型
  ├── LOW 媒体分辨率
  ├── 滑动窗口上下文压缩
  └── 自动语音活动检测
          │
          ▼
会话级保护
  ├── 45 秒空闲超时
  ├── 600 秒最长时限
  ├── 50,000 token 预算
  └── 最终 usage 汇总
```

## 7. 关键设计取舍

### 7.1 成本保护放在前后端两侧

只依赖前端限制并不可靠，因为客户端可能版本过旧、被修改或在网络恢复后集中发送数据。因此，前端负责尽早减少无效数据，后端负责执行不可绕过的输入、会话和预算边界。

### 7.2 优先保证信息新鲜度，而不是保证每帧送达

实时视觉问答中，最新画面通常比完整帧序列更有价值。项目因此选择丢弃或替换过时帧，不采用可靠但高积压的视频队列。这一方案同时改善回答相关性、延迟和成本。

### 7.3 使用运营上限，而不把它当作精确账单

token 预算基于 Gemini Live 返回的 usage metadata，在模型上报后才能判断是否达到阈值。因此，50,000 token 是会话保护线，不保证账单绝不超过该数值。真正的费用控制仍需要结合供应商账户配额、告警和账单监控。

### 7.4 不为首版引入收益不明确的本地智能

视觉变化检测、本地 VAD 和动态模型路由都有进一步降低调用量的潜力，但也会增加客户端计算、开发时间和误判风险。首版优先采用可解释、可测试的固定限制和队列策略。

## 8. 结论与后续方向

首版计划中的核心视觉语音对话流程已经完成，包括设备检测、实时多模态输入、语音与文字回答、用户打断、对话记录、异常恢复、终止解释和用量摘要。主要未完全实现项是实时翻转镜头；其他未实现功能基本属于设计时明确排除的范围。

运营成本控制最终形成了四层防线：前端减少无效媒体、后端限制输入与排队、Gemini 使用低分辨率和上下文压缩、逻辑会话执行超时与 token 预算。相比单纯降低采样率，这套方案也覆盖了异常连接、长会话、流量突发和不可控客户端等实际运营风险。

后续优化应建立在真实 usage 数据上，优先评估：

1. 根据连续帧差异自适应调整视频采样频率。
2. 按场景和问题复杂度选择不同模型或视觉输入频率。
3. 为服务端增加按用户、IP 或 API Key 的速率和并发限制。
4. 接入供应商配额、费用告警和按日成本看板。
5. 完成实时镜头翻转，并验证切换设备时不会产生重复媒体流。
